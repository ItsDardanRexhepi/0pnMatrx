# Module 05: Deploying Contracts

## The Contract Conversion Pipeline

One of the most powerful features of 0pnMatrx is the ability to describe a smart contract in plain English and have it converted, audited, and deployed automatically. This module walks through the entire pipeline.

The process follows five stages:

```
Plain English Description
    |
    v
Solidity Code Generation
    |
    v
Glasswing Security Audit
    |
    v
Deployment to Base Network
    |
    v
EAS Attestation (on-chain record)
```

## Step 1: Write a Plain-English Contract Description

Good contract descriptions are specific about behavior, roles, and conditions. Here is an example:

> Create a token called "MatrixCoin" with the symbol "MTRX". Total supply should be 1,000,000 tokens. The deployer receives the full initial supply. Include a burn function that lets any holder destroy their own tokens. Include a pause function that only the owner can use to stop all transfers in an emergency.

Notice the description specifies:
- **What** the contract is (an ERC-20 token)
- **Parameters** (name, symbol, supply)
- **Distribution** (deployer gets initial supply)
- **Functions** (burn, pause)
- **Access control** (pause is owner-only)

The more specific your description, the more accurate the generated contract.

## Step 2: Send It to Trinity

```bash
curl -X POST http://localhost:18790/chat \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{
    "message": "Create a token called MatrixCoin with symbol MTRX. Total supply 1000000. Deployer gets full supply. Include burn for any holder and owner-only pause for emergencies."
  }'
```

Trinity parses your intent and hands the structured specification to Neo. Neo invokes the contract generation service, which produces valid Solidity code based on your description. The generated code uses battle-tested OpenZeppelin libraries where applicable -- in this case, ERC20, ERC20Burnable, Pausable, and Ownable.

## Step 3: Understanding the Audit Report

Before deployment, the generated contract is automatically passed through **Glasswing**, the 0pnMatrx security auditing engine. Glasswing performs a 12-point vulnerability scan.

The audit report is included in Trinity's response:

```json
{
  "audit": {
    "status": "passed",
    "scanner": "glasswing",
    "vulnerabilities_found": 0,
    "checks_performed": 12,
    "severity_summary": {
      "critical": 0,
      "high": 0,
      "medium": 0,
      "low": 0,
      "informational": 1
    },
    "notes": [
      {
        "severity": "informational",
        "message": "Consider adding events for pause/unpause actions for off-chain monitoring"
      }
    ]
  }
}
```

Severity levels:
- **Critical / High**: Deployment blocked until resolved
- **Medium**: Deployment allowed but strongly recommended to fix
- **Low / Informational**: Suggestions for best practices

If the audit finds critical or high severity issues, the pipeline stops. Trinity will explain what was found and suggest corrections. You can modify your description and try again.

## Step 4: Deploy to Base Sepolia Testnet

Once the audit passes, Morpheus appears to confirm the deployment. This is an irreversible action -- once deployed, the contract exists permanently on the blockchain.

Morpheus presents:
- The contract that will be deployed
- The network (Base Sepolia testnet for testing, Base mainnet for production)
- Estimated gas cost
- The wallet that will own the contract

After you confirm, Neo deploys the contract and returns the transaction hash and contract address:

```json
{
  "deployment": {
    "network": "base-sepolia",
    "contract_address": "0x1234...abcd",
    "transaction_hash": "0xabcd...1234",
    "deployer": "0x5678...efgh",
    "block_number": 12345678
  }
}
```

Always deploy to **Base Sepolia** (testnet) first. Test your contract thoroughly before deploying to mainnet.

## Step 5: Reading the EAS Attestation

Every contract deployed through 0pnMatrx receives an **EAS (Ethereum Attestation Service) attestation**. This is an on-chain record that certifies:

- The contract was generated by 0pnMatrx
- It passed a Glasswing security audit
- The timestamp of deployment
- The audit results summary

The attestation UID is included in the deployment response:

```json
{
  "attestation": {
    "uid": "0xabc123...",
    "schema": "0pnMatrx-deployment-v1",
    "attester": "0x0pnMatrx-attestation-service",
    "data": {
      "contract": "0x1234...abcd",
      "audit_passed": true,
      "audit_score": 12,
      "timestamp": 1680000000
    }
  }
}
```

Anyone can verify this attestation on-chain. It serves as a trust signal -- users interacting with your contract can confirm it was professionally audited before deployment.

## Best Practices

- Always describe access control explicitly ("only the owner can...", "any holder can...")
- Deploy to testnet first, always
- Read the full audit report, including informational notes
- Keep your contract description focused -- one contract per request
- After deployment, verify the contract source code on the block explorer

## Key Takeaways

- Describe contracts in plain English with specific parameters and access control
- The pipeline: description, generation, Glasswing audit, deployment, attestation
- Critical/high audit findings block deployment
- Always deploy to Base Sepolia testnet before mainnet
- EAS attestations provide on-chain proof of audit
